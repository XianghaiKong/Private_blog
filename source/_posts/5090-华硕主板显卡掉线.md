## 事情起因

事情的起因发生在某天晚上运行深度学习代码时。按理说，这份代码所需的显存并不高，以 RTX 5090 的算力，理论上可以同时跑三个这样的实验。然而，就在代码启动的一瞬间，桌面直接“掉了”——显示器提示无信号输入。

经过后期的排查发现，其实是在那一瞬间，**独显停止了渲染桌面，系统自动切换到了核显**。但由于当时显示器的 DP 线是插在 5090 显卡上的，主板的 DP 口并没有接线，导致主板上的核显虽然接管了输出，但无法将画面传输给显示器，所以显示器表现为一直黑屏。

---

## 恢复显示器的正常输出

找到原因后，排查和恢复画面的第一步是让显示器重新亮起来：

1. **进入 BIOS 禁用核显**：将显示器线接到主板的 HDMI/DP 口，成功点亮并进入了 BIOS 界面。
2. **关闭核显设置**：在 BIOS 中将核显强制关闭。（华硕主板关闭核显的具体操作可参考 [这篇知乎教程](https://zhuanlan.zhihu.com/p/30736793224)）。
3. **重新接回独显**：设置完成后，将显示器线重新接回 5090 显卡，显示器恢复正常显示。

> ⚠️ **潜在隐患（注意事项）**：
> 这样操作后有一个小隐患：如果未来 5090 显卡真的发生硬件故障，再想进入 BIOS 页面就会比较麻烦。由于核显已被禁用，需要**扣掉主板上的纽扣电池静置五分钟（或短接 CLRTC 引脚）**，恢复主板出厂设置后，才能再次使用核显渲染出 BIOS 界面。

---

## 根本原因与解决方案

在解决了“黑屏”问题后，回到最初的起因：为什么跑代码瞬间会导致独显掉线？

初步推测，可能是由于代码开始跑的那一瞬间，**显卡启动功率过大（RTX 5090 的瞬时功耗极高，满载可达 600W 以上）**，导致电源的 OCP（过流保护）被触发，或是主板 PCIe 供电出现瞬时保护。

为了防止类似情况再次发生，可以采用以下几种方案：

### 方案一：限制显卡最大功耗（推荐）

在跑代码之前，可以通过 `nvidia-smi` 强制限制 5090 的最大功耗。例如，将其限制在 450W：

```bash
# 设置 GPU 功耗上限为 450W
sudo nvidia-smi -pl 450
```

设置完成后，再次输入 `nvidia-smi` 即可看到最大功率已经被限制。

**缺点**：每次重启电脑后该设置会失效。不过由于这台 5090 工作站安装的是 Ubuntu 系统，平时一直放在工位且基本不怎么关机，所以这个方案目前非常实用。*(注：如果需要开机自动生效，可以将其加入 `rc.local` 或设置为 systemd 开机服务。)*

### 方案二：代码层面的 GPU 预热 (Warm-up)

为了避免瞬间将 GPU 负载拉满，可以在代码中加入一些预热操作。例如在开始正式的大规模矩阵运算前，先用较小的 batch size 跑几个 step，让 GPU 的功耗平滑上升，避免产生巨大的瞬态电流。

## 遗留的疑问与探讨

最后，还有一个至今让我不太明白的问题：
工作站使用的是 **1300W 金牌电源**，按理说应对单张 5090 显卡是绰绰有余的，应该不至于无法为大功耗显卡供电。

> **💡 可能的解释（瞬态功耗 Power Excursion）：**
> 现代高端显卡在瞬间负载突变时，会产生极高的**瞬态功耗**，有时能达到 TDP 的 2 到 3 倍（持续几十微秒到几毫秒）。如果电源不是符合 ATX 3.0/3.1 标准的原生 PCIe 5.0 电源，其应对瞬态功耗的能力可能不足，从而瞬间触发电源的过载保护机制。

当然，这也仅仅是推测。具体的真正原因，只有留到后续有机会时再去深入探索和验证了。
